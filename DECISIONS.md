# Asymm Studio Architectural Decisions

| # | Decision | Why | Alternatives Considered |
|---|----------|-----|-------------------------|
| 1 | Keep `packages/sacred-geometry/` as the mathematical foundation and import or extend it instead of rewriting constants, easings, spacing, layouts, and color primitives. | The existing package is production-ready and prevents drift in PHI, Fibonacci, golden-angle, Vedic, and Kashmir Shaivism math. | Reimplement math inside each package; copy selected helpers into new modules; vendor a new design-token library. |
| 2 | Split the system into `asymm-aesthetic-engine`, `asymm-bio-resonance`, and `apps/lab-site`, with the lab app as the only place both engines meet. | This preserves dependency boundaries, keeps privacy-sensitive bio code isolated, and allows aesthetic work to evolve without pulling camera or identity concerns into it. | Cross-import packages directly; create a single monolithic engine; introduce a shared integration package before the public API stabilizes. |
| 3 | Map deterministic seeds to unit quaternions on S3 before deriving aesthetic tokens. | Quaternion space gives a compact continuous parameter domain that supports region classification, interpolation, and stable variation from numeric seeds. | Use raw PRNG values; derive tokens directly from string hashes; use 2D hue/spacing axes only. |
| 4 | Use FNV-1a content hashing, UTC-day temporal seeds, and explicit visitor fingerprints for seed inputs. | These inputs are deterministic, cheap, privacy-aware, and avoid collecting implicit visitor entropy. | Browser fingerprinting; random per-session seeds only; cryptographic hashing for every aesthetic seed. |
| 5 | Define eight named aesthetic regions as disjoint quaternion bounds with separate Design DNA docs. | Designers need semantic regions and editable briefs without touching seed math. | Single continuous unlabelled space; hardcoded theme names without bounds; one palette per component. |
| 6 | Derive color, typography, geometry, and motion tokens as pure functions from a quaternion. | Pure derivation makes tests deterministic, keeps rendering surfaces simple, and lets Astro/Svelte consumers share the same token contract. | Store generated tokens in JSON; compute tokens in components; use CSS-only randomization. |
| 7 | Generate palettes through a golden-angle hue walk with contrast checks. | The palette remains varied while preserving readable foreground/background pairings for WCAG AA goals. | Static palettes; random HSL offsets; port the Python calculator exactly without accessibility constraints. |
| 8 | Keep the 27-component vocabulary as catalog data and make variation preserve semantic structure. | The engine can change presentation while keeping accessible markup and component intent stable. | Generate arbitrary HTML strings; create separate code components for all 27 in V1; let each component own unrelated token logic. |
| 9 | Implement the alchemist transmuter as a conservative whitelist sanitizer that only permits generated `--asymm-*` style declarations. | It prevents XSS while still allowing deterministic engine styling to be emitted. | Use `innerHTML`; trust source markup; include a full HTML parser dependency and broad attribute passthrough. |
| 10 | Port bio-resonance numerical modules to strict TypeScript but relax `noUncheckedIndexedAccess` for dense numerical array code. | Strict boundaries remain, while ported math stays readable and avoids excessive index guards in hot numerical paths. | Keep the stricter option and litter code with non-null assertions; disable strict mode; rewrite algorithms around heavier typed containers. |
| 11 | Data-drive bio-resonance thresholds, kernel weights, frame caps, and stability limits. | Configurable thresholds make demos and future devices tunable without changing math internals. | Hardcode all constants from source material; expose mutable globals; infer thresholds from UI components. |
| 12 | Add CFL and `dt` bounds to PDE and engine evolution. | Numerical simulations must fail softly and avoid runaway values under frame drops or unusual inputs. | Trust caller timing; clamp only final rendered values; remove PDE evolution from V1. |
| 13 | Use real quaternion SLERP in particle evolution and clean GPU framebuffers on resize. | The source renderer had fake interpolation and resize leak risks; the port needed correctness and resource hygiene. | Keep approximate lerp; rebuild renderer from scratch; omit GPU lifecycle tests. |
| 14 | Put GPU/network forwarding behind an explicit bridge with a circuit breaker and health checks. | The privacy model forbids accidental camera or biometric transmission, so any IO path needs bounded behavior and clear ownership. | Let engine modules call `fetch`; remove GPU bridge entirely; silently retry forever. |
| 15 | Use `@noble/ed25519` and `@noble/hashes` for identity operations. | Identity must rely on audited primitives rather than ad hoc signing or hashing. | Browser `SubtleCrypto` only; custom crypto; bundled polyfills with unclear semantics. |
| 16 | Label VedicCrypto as `VedicObfuscation` and avoid encryption/decryption claims. | The method is numerological obfuscation, not cryptographic security, and the API should not mislead users. | Keep the original encryption naming; remove the module entirely; bury the caveat in docs only. |
| 17 | Store and transmit only a one-way presence hash, rejecting all-zero presence vectors. | Bio-resonance identity must avoid biometric persistence and prevent trivial empty-baseline identities. | Store raw calibration vectors; hash unvalidated data; persist camera-derived samples in local storage. |
| 18 | Centralize privacy footer copy for calibration and bio-resonance UI surfaces. | Every camera-facing page needs visible, consistent privacy language without copy drift. | Let each page author its own footer; rely on README privacy docs; show privacy only after permission is requested. |
| 19 | Rebuild Svelte bio-resonance components from scratch while keeping phase/state-machine ideas. | The source UI had heavy logging, hardcoded sizing, and raw implementation detail; the target needs Stripe-quality, accessible demos. | Directly port all Svelte files; wrap old components; defer UI until package APIs mature. |
| 20 | Camera-facing components own startup, teardown, error recovery, accessibility labels, reduced-motion CSS, and visible privacy copy. | The safest abstraction is for each demo surface to clean up streams and communicate status locally. | Centralize camera lifecycle in a global store; start camera automatically; rely on browser defaults for teardown. |
| 21 | Represent low-confidence PPG as `null` instead of returning misleading BPM values. | A false biometric reading is worse than no reading; downstream UI can show uncertainty honestly. | Return zero BPM; return the last value indefinitely; surface raw noisy estimates. |
| 22 | Scope Vitest to project-owned test globs instead of collecting dependency or read-only source-material tests. | The workspace contains reference material and dependencies that should not be treated as first-party tests. | Rename reference tests; run every discovered `*.test.*`; split the repo into separate test workspaces immediately. |
| 23 | Use Astro for lab pages and Svelte islands only where client interaction is needed. | Static pages stay fast and inspectable while calibration, gaze, and adaptive-reader demos hydrate as interactive islands. | Make the whole lab a SvelteKit app; ship plain static HTML only; hydrate every component globally. |
| 24 | Treat MediaPipe in V2 as alpha-depth IO with deterministic mock fallback. | Real camera/landmark integration should work where available, but demos and tests cannot depend on physical hardware. | Mock everything; require camera for all demos; fail if MediaPipe cannot load. |
| 25 | Maintain `DECISIONS.md` and `PORT_MAP.md` as root traceability artifacts. | Parallel tracks need a shared architectural memory and an auditable source-to-target migration record. | Scatter notes across ticket logs; keep decisions only in commit messages; rely on source comments for migration history. |
| 26 | Test Astro primitives with `@astrojs/compiler` parsing plus token-contract assertions instead of a bespoke renderer. | It validates real Astro syntax inside the package while keeping tests deterministic and independent of a browser runtime. | Snapshot generated HTML through a lab app build; use string-only tests; add a full Astro test harness per package. |
| 27 | Keep each Astro primitive compositionally independent even when their token setup is similar. | Ticket 2 explicitly requires no cross-dependencies, and isolated primitives are easier for designers to copy into demos. | Create a shared Astro token wrapper; generate primitives from a factory; make Batch 2 import Batch 1 components. |
| 28 | Mirror the Astro primitive CSS-variable contract inside the Svelte SeedExplorer instead of importing Astro components into the client island. | Astro components are server-rendered primitives, while the explorer needs live client-side updates without page reloads. | Hydrate each Astro primitive separately; generate HTML strings in Svelte; skip live previews and show tokens only. |
| 29 | Route FaceMesh creation through a small live/mock detector factory with injectable live loading for tests. | MediaPipe needs real browser IO in alpha environments while CI and no-camera demos need deterministic behavior through the same landmark pipeline. | Keep FaceMeshIntegration mock-only; initialize MediaPipe directly inside components; require camera permission before package APIs can be constructed. |
| 30 | Use explicit region palette presets in the comparison island rather than trying to infer a representative seed for each region. | The A/B page must make differences immediately obvious and compare named region intent, not nearest random seed samples. | Search for a seed inside each region; reuse documentation JSON at runtime; show only token tables instead of rendered panels. |
| 31 | Persist the active lab theme in `sessionStorage` and apply it as root CSS variables. | V2 explicitly wants navigation-scoped persistence without long-lived visitor storage, and root variables make the swap instant. | Use `localStorage`; keep the selection in memory only; route every page through server-side region parameters. |
| 32 | Cover the mixed TS/Astro aesthetic package with TS sub-barrel imports plus static primitive export checks. | Vitest's Node path cannot parse `.astro` component modules from the top-level barrel, but the public API still needs regression coverage. | Import the root barrel directly in Vitest; drop primitive export coverage; add a browser-only test harness before package polish. |
| 33 | Keep `UnifiedSignalExtractor` pure and add live/mock pipeline adapters around it. | Camera, canvas, MediaPipe, and simulation concerns belong at the IO boundary while signal math stays deterministic and testable. | Put MediaPipe logic inside the extractor; make Svelte components assemble landmarks and frames manually; keep PPG simulation only in UI code. |
| 34 | Let calibration request live camera first and fall back to the same visual ritual with `MockSignalPipeline`. | The alpha demo should be compelling without hardware while making the live/privacy state explicit when permission is granted. | Require camera before starting; ship a separate mock-only page; hide measurements until real MediaPipe is available. |
| 35 | Smooth gaze cursor movement by projecting screen points into unit quaternions and applying SLERP. | The project already uses quaternion motion semantics, and SLERP gives a visibly calmer cursor than direct coordinate assignment. | Linear interpolation in x/y; CSS-only transition smoothing; keep the previous scripted cursor without real pipeline input. |
| 36 | Drive the stress reader from a visible HRV/coherence trace rather than an invisible scalar. | Users need to see why density, color, and motion are changing, especially in simulated mode. | Keep only a stress meter; adapt silently from coherence; make the reader purely decorative with no graph. |
| 37 | Ship continuous auth as a slotted CSS guard with an explicit DOM-content limitation. | The alpha pattern demonstrates presence-aware hiding while avoiding a false security claim about client-rendered content. | Remove content from the DOM; build only an overlay; present the demo as real data security. |
| 38 | Test the bio package public API through TS sub-barrels and static Svelte barrel checks. | Vitest can cover the math, pipeline, MediaPipe, identity, and calibration contracts directly while avoiding Svelte runtime compilation in Node. | Import the root barrel and execute `.svelte` components in Vitest; skip component export coverage; move all API checks into the lab build. |
| 39 | Split the lab site foundation into reset, tokens, and typography with a self-hosted Asymm Sans font and Fibonacci/PHI variables. | The lab needs one inspectable design substrate for later polish tickets without external font requests or scattered arbitrary spacing. | Keep a single global stylesheet; import Google Fonts; let each component define its own spacing and type scale. |
| 40 | Polish bio component visuals through token-aware CSS fallbacks instead of moving package components into the lab site's stylesheet. | The components must remain reusable outside the lab while still inheriting the lab foundation when embedded there. | Hardcode the lab theme into package components; make lab pages wrap every component in additional CSS; refactor all component styles into a shared package CSS file. |
| 41 | Generate lab navigation, active states, mobile menu links, and breadcrumbs from the current Astro URL and a single nav array. | A shared route source prevents desktop/mobile drift and keeps each page keyboard navigable without per-page layout code. | Hardcode links twice; add client-side routing state; make every page declare its own breadcrumbs. |
| 42 | Make every lab page start with a purpose statement and contextual next links. | The demo site needs to explain why each surface exists before asking for interaction or camera trust. | Keep pages as demo embeds only; move orientation into the manifesto; add a separate documentation page for all explanations. |
| 43 | Add a Playwright responsive-a11y audit for lab routes outside the Vitest glob. | Ticket 19 needs executable browser checks across fixed widths without interfering with package unit tests. | Keep the audit manual; fold browser checks into Vitest; rely only on Astro build output. |
| 44 | Compose aesthetic regions with bio calibration through lab-owned CSS variables in `AestheticAuth`. | The app is the only allowed meeting point between packages, and CSS variables let the ritual change style without importing aesthetic code into bio-resonance. | Add aesthetic imports inside bio components; duplicate calibration per region; make the global theme switcher the only integration proof. |
| 45 | Drive the lab index theme from `temporalSeed()` in a client island with yesterday/today/tomorrow previews. | The daily aesthetic should be visible and interactive without storing visitor state or changing package boundaries. | Compute the date only at build time; persist a daily theme in session storage; add server-side routing per day. |
| 46 | Build the integration showcase from lab-owned composition wrappers instead of modifying package internals. | The page needs to demonstrate cross-engine behavior while preserving the dependency rule that packages meet only in `apps/lab-site`. | Add integration APIs inside both packages; duplicate demos as static screenshots; make a single overloaded Svelte component. |
